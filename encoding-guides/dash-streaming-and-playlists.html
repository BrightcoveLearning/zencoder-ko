소개

Dash는 Zencoder를 사용하는 라이브 스트림과 호환되지 않습니다.

DASH는 특정 비디오 형식, 코덱 또는 데이터에 구애받지 않는 패키징 형식 (예: Zip) 이 아니기 때문에 미디어를 지정하는 새로운 옵션을 도입했습니다. streaming_delivery_format파일은 특정 표준에 따라 구조화/형식이 지정되어야 합니다. “dash”를 로 선택하면 DASH와의 호환성에 대한 몇 가지 기본 옵션이 수정되고 출력 미디어 파일을 DASH 요구 사항과 일치하는 디렉토리 구조에 넣을 수 있습니다. streaming_delivery_format DASH 미디어 출력도 미디어가 DASH 스트리밍을 위해 준비되도록 “세그먼트”로type설정해야 합니다.

{
    "input": "http://s3.amazonaws.com/zencodertesting/test.mov",
    "outputs": [
    {
        "streaming_delivery_format": "dash",
        "video_bitrate": 700,
        "type": "segmented",
        "url": "s3://mybucket/dash-examples/sbr/rendition.mpd"
    }
    ]
}

“dash”로streaming_delivery_format설정된 출력에 대한 세부 정보:

  • fixed_keyframe_interval true로 설정됩니다.
  • segment_seconds기본값은 2입니다.
  • 기본 형식은 다음과 같습니다m4f ( mpd매니페스트 파일 사용)

세그먼트화된 DASH 출력을 위한 파일 구조:

  • mpd매니페스트 파일은 출력의 루트 레벨에 있습니다.
  • 오디오 및 비디오 미디어에는 각각 고유한 하위 디렉토리 ("" 및audio/und "“video/1 ") 가 있습니다.
  • 이러한 각 디렉토리에는 초기화 데이터가 있는init.mp4 "" 가 있습니다.
  • 미디어 데이터는 " seg-N.m4f " 파일에 저장됩니다 (여기서 N은 0에서 시작하여 각 세그먼트에 대해 증가합니다).

“on_demand”로streaming_delivery_profile설정된 DASH 출력은 세그먼트가 아닌 조각화된 미디어 파일을 사용합니다.

  • mpd매니페스트 파일은 출력의 루트 레벨에 있습니다.
  • 오디오 및 비디오 미디어는 각각 매니페스트와 동일한 디렉토리에있는 단일 파일입니다.
  • 미디어 파일의 이름은 매니페스트와 유사하게 지정되지만 적절한 경우-audio "" 또는-video "" 접미사가 추가됩니다.

지원되는 DASH 구성:

  • 지원되는 형식: m4f및webm
  • 지원되는 비디오 코덱: h264hevc , vp8 , 및vp9
  • 지원되는 오디오 코덱: aacac3 , eac3 , 및vorbis
  • " webm " 형식의 출력은 현재 “on_demand”만 지원합니다. streaming_delivery_profile
  • " webm " 형식의 출력은 현재 DRM을 지원하지 않습니다.

다중 비트 레이트

Zencoder에서 생성 한 DASH 출력은 자체적으로 재생되는 데 필요한 구조를 갖지만 이러한 스트리밍 전송 옵션의 이점을 제대로 활용하려면 변환을 나열하는 "재생 목록"유형 출력과 함께 다른 비트 전송률 (변환)로 여러 버전을 생성해야합니다. 플레이어가 현재 네트워크 상태에 따라 필요에 따라 비트 전송률을 높이거나 낮출 수 있습니다. 재생 목록 출력은 종속 출력입니다. 즉, 레이블로 다른 출력을 참조하고 해당 출력의 데이터를 사용하여 재생 목록의 최종 표현을 생성합니다. 재생 목록 출력에서 스트림 경로를 설정하여 변환 파일의 위치에 대한 상대 경로가되도록주의해야합니다.

{
    "input": "http://s3.amazonaws.com/zencodertesting/test.mov",
    "outputs": [
    {
        "label": "dash-1200",
        "streaming_delivery_format": "dash",
        "video_bitrate": 1200,
        "type": "segmented",
        "url": "s3://mybucket/dash-examples/mbr/1200k/rendition.mpd"
    },
    {
        "label": "dash-700",
        "streaming_delivery_format": "dash",
        "video_bitrate": 700,
        "type": "segmented",
        "url": "s3://mybucket/dash-examples/mbr/700k/rendition.mpd"
    },
    {
        "label": "dash-400",
        "streaming_delivery_format": "dash",
        "video_bitrate": 400,
        "type": "segmented",
        "url": "s3://mybucket/dash-examples/mbr/400k/rendition.mpd"
    },
    {
        "streaming_delivery_format": "dash",
        "type": "playlist",
        "url": "s3://mybucket/dash-examples/mbr/manifest.mpd",
        "streams": [
        { "source": "dash-1200", "path": "1200k" },
        { "source": "dash-700",  "path": "700k" },
        { "source": "dash-400",  "path": "400k" }
        ]
    }
    ]
}

암호화

Zencoder는 또한 Marlin 또는 Playready 라이선스 서버와 통합하는 데 필요한 헤더와 함께 CENC 암호화 DASH 출력 생성을 지원합니다. 각 출력에 대해 다음을 포함하여 drm 옵션의 해시를 지정하기만 하면 됩니다. methodkey_id , 및content_key . key_id및 는 DRM 공급업체에서 검색되거나 미리 생성되어 DRM 공급업체로content_key전송되어야 합니다. 이러한 키가 있으면 단일 변환에 대한 Zencoder API 구조는 다음과 같습니다.

트랜스 먹싱

물론 비디오의 여러 형식을 생성하는 것은 비용이 많이 들고 낭비가 될 수 있습니다. 따라서 Zencoder는copy_video및 를 사용하여 동일한 인코딩된 스트림을 다른 형식 (전송) 으로 재패키징할 수 있도록 지원합니다. copy_audio옵션. 비디오가 인코딩되지 않고 트랜스 먹싱 된 출력은 일반 재생 시간의 1/4로 청구됩니다. 따라서 동일한 비디오 (예: MP4, HLS 및 DASH) 의 여러 형식을 만들 계획이라면 먼저 각 비트 전송률의 MP4 버전을 인코딩한 다음 해당 출력을 트랜스미스하여 HLS 및 DASH 버전을 만들 수 있습니다. (비디오가 더 이상 단일 스트림이 아니기 때문에 분할 된 형식에서 트랜스 먹싱을 지원하지 않습니다.)

후속 세그먼트 스트리밍 형식의 제약 조건에 따라 MP4가 준비되도록 하려면 MP4 출력의prepare_for_segmenting옵션을 사용하십시오. 이렇게하면 나중에 세그먼트 형식으로의 트랜스 먹싱과 호환되도록 MP4의 비디오 스트림을 인코딩 할 때 키 프레임 및 캡션에 대한 기본값을 제어합니다.

MP4를 트랜스 먹싱 된 출력의 소스로 사용하면 재생 목록 출력과 유사하게 작동합니다. 다른 출력의 레이블과 일치하는 소스 옵션을 지정하지만 이번에는 스트림 해시 대신 출력에 대한 기본 수준 옵션의 일부입니다.

{
    "input": "http://s3.amazonaws.com/zencodertesting/test.mov",
    "outputs": [
    {
        "label": "mp4-700k",
        "prepare_for_segmenting": ["hls", "dash"],
        "video_bitrate": 700,
        "url": "s3://mybucket/transmux-examples/sbr/video.mp4"
    },
    {
        "source": "mp4-700k",
        "copy_video": true,
        "copy_audio": true,
        "type": "segmented",
        "url": "s3://mybucket/transmux-examples/sbr/hls/video.m3u8"
    },
    {
        "source": "mp4-700k",
        "copy_video": true,
        "copy_audio": true,
        "streaming_delivery_format": "dash",
        "type": "segmented",
        "url": "s3://mybucket/transmux-examples/sbr/dash/video.mpd"
    }
    ]
}

배달 최적화

DASH는 스트리밍 전송을위한 것이므로 플레이어가 불필요하게 낮은 비트 전송률 인코딩으로 전환하지 않도록 각 변환에서 비트 전송률의 최고치를 제어하는 것이 좋습니다. 디코더 / 플레이어가 처리 할 수있는 것에 따라 인코딩 된 스트림의 가변성을 제어하는 옵션은 다음과 같습니다.

  • decoder_bitrate_cap
  • decoder_buffer_size
참고: max_video_bitrate전체 스트림에 대한 최대 평균 비트 전송률을 제어하는 데 사용하며 스트리밍 콘텐츠와 함께 사용하기에 적합하지 않습니다.

일반적인 권장 사항은 비디오decoder_bitrate_cap스트림에 사용할 수 있는 최대 비트 전송률로 설정하는 것입니다. 오디오 및 파일 형식에 대한 약간의 오버 헤드를 고려해야합니다. decoder_buffer_size는 기술적으로 디코딩 디바이스에서 사용 가능한 비디오 버퍼 메모리와 일치하도록 되어 있지만, 미리 버퍼링되는 콘텐츠의 양을 제어하는 데 더 자주 사용됩니다. 예를 들어 500 (초당 킬로비트) decoder_bitrate_cap인 경우 의 배수로 작동합니다. decoder_bitrate_capdecoder_buffer_size 1000 (Kilobits) 이었고 버퍼는 2 초 분량의 비디오를 저장할 수 있습니다. 엔코더를 허용하는 보다video_bitrate약간 낮게 설정하는 것이 좋습니다. decoder_bitrate_cap더 많은 데이터를 사용하여 복잡한 씬을 인코딩하고 간단한 씬에서는 더 적은 데이터를 인코딩할 수 있습니다. 적절한 시작점은 파일 형식에 대해 최대 10% 의 오버 헤드를 가정하고video_bitrate 10% 를 보다 낮게 설정하고 약 1을 허용하는 것입니다. decoder_bitrate_cap 5초 동안 버퍼링할 비디오입니다.

고객이 1000Kbps 연결을 사용할 수 있다고 가정합니다.

  • 1000 * 0.90 = 형식 오버 헤드를 위해 10 %를 뺀 후 900Kbps 사용 가능.
  • 900 - 128 = 오디오에 대해 128킬로바이트를 뺀decoder_bitrate_cap후 772킬로바이트의 비디오입니다.
  • 772 * 1.5 = 1158Kb decoder_buffer_size (1.5 초 분량의 비디오).
  • 772 * 0.90 = 695 Kbps의 타겟 비디오 비트레이트 (보다 10% 낮음decoder_bitrate_cap ).

따라서 더 정확한 스트리밍 인코딩 예를 보려면 다음을 수행하십시오.

{
    "input": "http://s3.amazonaws.com/zencodertesting/test.mov",
    "outputs": [
    {
        "label": "mp4-1000k",
        "prepare_for_segmenting": ["hls", "dash"],
        "audio_bitrate": 128,
        "decoder_bitrate_cap": 772,
        "decoder_buffer_size": 1158,
        "video_bitrate": 695,
        "url": "s3://mybucket/transmux-examples/sbr/video.mp4"
    },
    {
        "source": "mp4-1000k",
        "copy_video": true,
        "copy_audio": true,
        "type": "segmented",
        "url": "s3://mybucket/transmux-examples/sbr/hls/video.m3u8"
    },
    {
        "source": "mp4-1000k",
        "copy_video": true,
        "copy_audio": true,
        "streaming_delivery_format": "dash",
        "type": "segmented",
        "url": "s3://mybucket/transmux-examples/sbr/dash/video.mpd"
    }
    ]
}
참고: 위의 ~ 아니다 HLS 인코딩이 Apple의 검증을 통과하도록 보장하기에 충분합니다. 1000Kbps 연결에서 원활하게 스트리밍되지만 Apple은 버퍼링 측면을 고려하지 않고 세그먼트 당 비트 전송률 (짧은 기간이있을 수있는 최종 세그먼트 포함)을 측정합니다. HLS 인코딩이 Apple의 유효성 검사를 통과하도록 하려면 HLS 가이드를 참조하십시오.
